# Technical Audit — Backup Banner Generator

**Date:** 2026-06-26
**Auditor:** Claude (Sonnet 4.6)
**Codebase:** `/Users/filip/Projects/_Internal/Rive/riveBackupImageGenerator`
**Live version:** https://bbg.mrwolf.be

---

## Overview

The Backup Banner Generator (BBG) is a well-conceived, functionally complete tool for generating JPEG backup images from HTML5 banner creatives. It runs in two modes: a **CLI** (batch-process a local `input/` folder) and a **web UI** (drag-and-drop upload with async job tracking). A secondary **Ad Validator** subsystem analyses creatives against platform presets.

**Overall health score: 7.5 / 10**

The core pipeline (upload → extract → capture → compress → deliver) is solid. The score is held back by in-memory-only job state, unbounded Chromium spawning in the validator, a CSP `'unsafe-inline'` weakness, and a handful of missing reliability safeguards — all fixable without fundamental redesign.

## Findings

Each finding uses the format: **Problem · Evidence · Risk · Recommended fix · Priority · Effort**

### Security

---

#### SEC-1 — CSP allows `'unsafe-inline'` in `script-src`

**Problem:** The Content Security Policy header includes `script-src 'self' 'unsafe-inline'`, which defeats the XSS protection that CSP is meant to provide.

**Evidence:** `src/webServer.js:968`
```js
res.setHeader('Content-Security-Policy',
  "default-src 'self'; script-src 'self' 'unsafe-inline'; ...");
```
The restriction exists because all frontend JS is currently in one inline `<script>` block in `index.html`.

**Risk:** High. If any user-controlled data were ever reflected into the page (error messages, filenames), an attacker could inject and execute arbitrary JS, which could then steal the localStorage auth session.

**Recommended fix:** Move the inline `<script>` block to `src/public/app.js`, reference it with `<script src="app.js">`, and remove `'unsafe-inline'` from the CSP.

**Priority:** High | **Effort:** 2–4 hours

---

#### SEC-2 — Login password comparison is not timing-safe

**Problem:** `handleLogin` uses `===` to compare passwords, which is vulnerable to timing side-channel attacks.

**Evidence:** `src/webServer.js:863`
```js
if (username !== adminUser || password !== adminPass) {
```

**Risk:** Medium. The rate limiter (30 req/min) makes this impractical to exploit, but it is a correctness issue.

**Recommended fix:**
```js
const usernameOk = crypto.timingSafeEqual(Buffer.from(username), Buffer.from(adminUser));
const passwordOk = crypto.timingSafeEqual(Buffer.from(password), Buffer.from(adminPass));
if (!usernameOk || !passwordOk) { ... }
```

**Priority:** Medium | **Effort:** 30 minutes

---

#### SEC-3 — Malicious `.riv` file can execute arbitrary JS in Playwright ⚠️

**Problem:** The `.riv` processing path copies the uploaded file as a `.js` file and serves it to a Playwright browser. There is no binary format validation — any `.js` payload renamed to `.riv` will be executed.

**Evidence:** `src/webServer.js:363–380`
```js
const jsFile = path.join(creativeDir, `${sanitized}.js`);
await fs.copyFile(file.path, jsFile);
// ... then served to Playwright which loads it as JS via the Rive runtime
```
Playwright is launched with `--disable-web-security`, meaning the injected JS can reach any localhost port, the AWS/GCP metadata service, and internal network endpoints.

**Risk:** High for any multi-user or public deployment. Low for private single-user deployment.

**Recommended fix:** Validate `.riv` files by checking their binary magic bytes before processing. Rive files have a known binary header. Reject files that fail validation.

**Priority:** High | **Effort:** 2 hours

---

#### SEC-4 — Auth session stored in `localStorage` — XSS-accessible

**Problem:** The frontend stores auth identity (userId, tenantId, clientId) in `localStorage` with a 14-day TTL.

**Evidence:** `src/public/index.html:142–169`

**Risk:** Medium. In combination with SEC-1, a successful XSS could steal auth tokens. Resolving SEC-1 substantially reduces this risk.

**Recommended fix:** Once SEC-1 is fixed and `'unsafe-inline'` is removed, this risk drops to low. As defence-in-depth, `HttpOnly`/`SameSite=Strict` session cookies would be more secure but require auth architecture changes.

**Priority:** Medium (after SEC-1) | **Effort:** 3–4 hours

---

#### SEC-5 — Rate limiter `Map` grows unbounded

**Problem:** The in-memory rate limiter stores timestamps per IP in a `Map` that is never pruned of stale entries. Sustained traffic from many unique IPs grows this Map indefinitely.

**Evidence:** `src/webServer.js:1002–1019`

**Risk:** Low-medium. Requires millions of unique IPs before it becomes a real problem, but stale entries never expire.

**Recommended fix:** Prune entries older than the rate limit window (60 s) on each access, or replace with `express-rate-limit`.

**Priority:** Medium | **Effort:** 1 hour

---

#### SEC-6 — Health endpoint exposes operational metrics without authentication

**Problem:** `GET /api/v1/health` is registered before auth middleware and is public. It returns job counts, auth mode, and all metrics counters/timings.

**Evidence:** `src/webServer.js:876–890`, `src/webServer.js:1030`

**Risk:** Low. Leaks auth mode (`production` vs `development`), active job count, and upload rejection rates.

**Recommended fix:** Strip sensitive fields from the public response (only return `status`, `version`, `uptime`). Move detailed metrics to an authenticated `/api/v1/metrics` endpoint.

**Priority:** Low | **Effort:** 30 minutes

---

#### SEC-7 — Playwright launched with `--disable-web-security`

**Problem:** All Chromium instances start with `--disable-web-security --disable-features=IsolateOrigins,site-per-process`, disabling SOP, CORS, and CSP for every rendered page.

**Evidence:** `src/browserPool.js:85–91`

**Risk:** Medium. Intentional for Rive CDN loading. Any JS in the banner context (including SEC-3 payloads) can reach any localhost port or cloud metadata endpoints.

**Recommended fix:** Bundle `rive.js` locally so cross-origin loading is unnecessary, then remove the `--disable-web-security` flag.

**Priority:** Medium | **Effort:** 1 day

---

#### SEC-8 — No binary format validation for `.riv` and video file uploads

**Problem:** Only ZIP files receive magic-byte validation. `.riv` and video files are accepted based on extension alone.

**Evidence:** `src/webServer.js:547–558` — ZIP check present, no equivalent for other types.

**Risk:** Low-medium. A misnamed file either fails during processing or (for `.riv`) executes as JS.

**Recommended fix:** Add magic-byte checks for video formats (MP4: `ftyp`, WebM: `\x1a\x45\xdf\xa3`) and Rive binaries.

**Priority:** Medium | **Effort:** 2 hours

### Code Quality

---

#### CQ-1 — `webServer.js` is a 1,113-line monolith

**Problem:** The server file contains route registration, job lifecycle logic, Multer config, cleanup, auth wiring, rate limiting, and a custom `Semaphore` class.

**Evidence:** `src/webServer.js` — 1,113 lines.

**Risk:** High maintenance cost; harder to test in isolation.

**Recommended fix:** Extract `processJob` + job lifecycle into `src/jobs/jobProcessor.js`, rate limiter into `src/rateLimiter.js`, Multer config into `src/upload.js`.

**Priority:** Medium | **Effort:** 2–3 days

---

#### CQ-2 — `APP_VERSION` is hardcoded

**Problem:** `const APP_VERSION = '1.0.0'` will drift out of sync with `package.json`.

**Evidence:** `src/webServer.js:41`

**Recommended fix:** Read from `package.json` at startup.

**Priority:** Low | **Effort:** 15 minutes

---

#### CQ-3 — Utility functions duplicated inside `webServer.js`

**Problem:** `parseNumberEnv`, `parsePortValue`, `boolEnv` are defined inline in `webServer.js` despite `config.js` existing for this purpose.

**Evidence:** `src/webServer.js:44–64`, `src/config.js`

**Recommended fix:** Move to `config.js` and import.

**Priority:** Low | **Effort:** 30 minutes

---

#### CQ-4 — `assertOwnership` and `assertValidatorOwnership` are identical functions

**Evidence:** `src/webServer.js:216–230`

**Recommended fix:** One generic `assertJobOwnership(job, auth, res, label)` function.

**Priority:** Low | **Effort:** 15 minutes

---

#### CQ-5 — `validatorWorkRoot()` ignores its argument

**Problem:** Called as `validatorWorkRoot(job.id)` but the function takes no parameters.

**Evidence:** `src/webServer.js:139–144` (definition), `src/webServer.js:673` (call site).

**Recommended fix:** Remove the argument from the call site.

**Priority:** Low | **Effort:** 5 minutes

---

#### CQ-6 — `shouldIgnore` uses over-broad string includes for `__MACOSX`

**Problem:** `entryName.includes('__MACOSX')` would incorrectly drop a legitimately named file like `my__MACOSX__folder/sprite.png`.

**Evidence:** `src/extractZip.js:11–23`

**Recommended fix:** Use `entryName.startsWith('__MACOSX/')` or an exact match.

**Priority:** Low | **Effort:** 5 minutes

---

#### CQ-7 — `isPathSafe` has a confusing exported API

**Problem:** The function first checks `path.resolve(targetPath) !== targetPath` — silently returning `false` for any relative path. Works correctly at call sites (where paths are pre-resolved) but is misleading as a utility.

**Evidence:** `src/extractZip.js:25–29`

**Recommended fix:** Assert the path must be absolute, or remove the misleading early return.

**Priority:** Low | **Effort:** 10 minutes

---

#### CQ-8 — `riveTemplate.js` hardcodes a CDN URL with no integrity check

**Problem:** The generated Rive HTML always loads from `https://s0.2mdn.net/creatives/assets/5617025/rive.js`. If this URL changes or is compromised, all Rive captures fail or execute malicious JS.

**Evidence:** `src/riveTemplate.js:44`

**Recommended fix:** Bundle `rive.js` locally. At minimum, add a Subresource Integrity (SRI) hash and intentionally pin the version.

**Priority:** Medium | **Effort:** 2 hours

---

#### CQ-9 — `Semaphore` class duplicates `BrowserPool` waiter logic

**Evidence:** `src/webServer.js:89–109` vs `src/browserPool.js`

**Recommended fix:** Extract `Semaphore` into `src/semaphore.js` and import in both.

**Priority:** Low | **Effort:** 30 minutes

---

#### CQ-10 — Download callback cleans up job on any response, including aborted downloads

**Problem:** `res.download(path, name, callback)` fires its callback on success *and* failure/disconnect. An interrupted download permanently deletes the result.

**Evidence:** `src/webServer.js:792–798`
```js
res.download(zipPath, `backup-images-${job.id}.zip`, async () => {
  await cleanupJob(job);
});
```

**Risk:** Medium. User cannot retry a failed download; must re-upload and reprocess.

**Recommended fix:** Remove cleanup from the download callback; rely on the existing 30-minute TTL-based cleanup instead.

**Priority:** Medium | **Effort:** 1 hour

### Performance & Reliability

---

#### PERF-1 — `renderChecks.js` launches a fresh Chromium per validation, bypassing the browser pool ⚠️

**Problem:** Every call to `checkRenderability` does `chromium.launch()` / `browser.close()` outside the `BrowserPool`. Under concurrent validator jobs, dozens of Chromium processes can launch simultaneously.

**Evidence:** `src/validator/checks/renderChecks.js:35`
```js
browser = await chromium.launch({ headless: true });
```

**Risk:** High. On resource-constrained containers (the Coolify deployment), concurrent validations could crash the server.

**Recommended fix:** Pass a reference to `getBrowserPool()` into `checkRenderability`, or add a separate semaphore to cap concurrent validation instances.

**Priority:** High | **Effort:** 2–3 hours

---

#### PERF-2 — Local file server always starts at port 3002 with recursive fallback

**Problem:** `createServer(3002, serveDir)` tries port 3002 and on `EADDRINUSE` recursively tries 3003, 3004, etc. Multiple concurrent jobs race on the same starting port.

**Evidence:** `src/localServer.js:31–43`

**Recommended fix:** Use `app.listen(0, 'localhost', ...)` — port 0 lets the OS assign a free port instantly.

**Priority:** Medium | **Effort:** 30 minutes

---

#### PERF-3 — Jobs are stored in memory only

**Problem:** `InMemoryJobStore` holds all state in a JavaScript `Map`. On server restart, all jobs (including completed ones awaiting download) are lost.

**Evidence:** `src/jobs/JobStore.js:27`

**Risk:** Medium. Users who are processing or waiting to download when a restart occurs lose their work.

**Recommended fix:** The `JobStore` interface is already abstracted. A SQLite-backed store would survive restarts with minimal changes.

**Priority:** Medium | **Effort:** 1–2 days

---

#### PERF-4 — Stuck `processing` jobs are never cleaned up

**Problem:** `pruneStaleSessions` explicitly skips `status === 'processing'` jobs. A job that crashes mid-flight is never pruned.

**Evidence:** `src/webServer.js:270`
```js
if (s.status === 'processing') continue;
```

**Recommended fix:** Add a secondary TTL for `processing` jobs (e.g., 30 minutes), after which they transition to `error` and are cleaned up.

**Priority:** Low | **Effort:** 1 hour

---

#### PERF-5 — AdmZip builds output ZIP synchronously in memory

**Problem:** The result ZIP is assembled with `new AdmZip()` / `addLocalFile()` / `writeZip()` — all file contents held in memory during build.

**Evidence:** `src/webServer.js:449–481`

**Risk:** Low at current scale (~50 × 80 KB = 4 MB). Could become significant at larger batch sizes.

**Recommended fix:** For future scale, replace with streaming `archiver`.

**Priority:** Low | **Effort:** 4 hours

---

#### PERF-6 — No per-user concurrency limit

**Problem:** One user submitting 50 files can occupy all 3 browser semaphore slots, blocking other users.

**Evidence:** `src/webServer.js:111` — single global semaphore.

**Recommended fix:** Add a per-tenantId semaphore in addition to the global cap.

**Priority:** Low | **Effort:** 2–3 hours

---

#### PERF-7 — Browser pool does not handle Chromium crashes

**Problem:** If a pooled browser is OOM-killed, the pool returns a dead instance to the next caller.

**Evidence:** `src/browserPool.js:14–57` — no crash detection.

**Recommended fix:** Listen for `browser.on('disconnected')`, discard the dead instance, and allow re-launch.

**Priority:** Medium | **Effort:** 2 hours

### UX

---

#### UX-1 — Validator mode only visible on localhost

**Problem:** `initModeSwitch()` hides the mode switch unless `isLocalhost()` is true. The live site `bbg.mrwolf.be` shows no validator UI even though all API routes are deployed.

**Evidence:** `src/public/index.html:538–546`

**Recommended fix:** If the validator is production-ready, remove the guard. If it's beta, document it clearly.

**Priority:** Medium | **Effort:** 5 minutes (to expose) or aligned with validator readiness

---

#### UX-2 — No file size shown in the uploaded files list

**Risk:** Users get no visual hint that a file is near the 200 MB limit until the upload fails.

**Recommended fix:** Show file size in the file row; warn if approaching the limit.

**Priority:** Low | **Effort:** 1 hour

---

#### UX-3 — Download triggers immediate cleanup even on network failure

See CQ-10. A failed or interrupted download permanently destroys the result ZIP.

**Priority:** Medium | **Effort:** 1 hour

---

#### UX-4 — No way to cancel in-progress processing

**Recommended fix:** Add `DELETE /api/v1/jobs/:jobId` and a Cancel button.

**Priority:** Low | **Effort:** 2–3 hours

---

#### UX-5 — Upload limits not shown in the drop zone

**Recommended fix:** Add "max 200 MB per file · max 50 files" to the drop zone hint.

**Priority:** Low | **Effort:** 5 minutes

---

#### UX-6 — Polling uses fixed interval with no backoff

**Problem:** Backup polling is 1 s, validator is 500 ms — no backoff for longer jobs.

**Recommended fix:** Exponential backoff, starting at 500 ms, capping at 3–5 s.

**Priority:** Low | **Effort:** 1 hour
